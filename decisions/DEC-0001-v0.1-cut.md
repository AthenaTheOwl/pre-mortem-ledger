# DEC-0001 — v0.1 control-plane cut

- **Status:** accepted
- **Date:** 2026-06-22
- **Spec:** [`specs/0002-design/`](../specs/0002-design)
- **Ledger row:** `2026-M06-initialization` in `data/ledger/runs.jsonl`

## Context

The Pre-Mortem Ledger is monthly tooling that scores the largest
concentrated investing position against three to five named failure
modes. The v0.1 cut had to deliver a runnable substrate — schema, CLI,
ledger, rubric, methodology — before the first real position could be
written up. The choices below shaped that cut.

## Decisions

### 1. JSON Schema is the structural source of truth, Pydantic mirrors it

`schemas/premortem.schema.json` is the cross-language contract.
`pre_mortem_ledger.schema` is a Pydantic mirror, pinned to the JSON
Schema by a round-trip test in `tests/test_schema.py`. Drift between
the two is caught at test time, not at render time.

**Rejected:** treating the Pydantic model as primary and generating
JSON Schema from it. Reason: future readers (jq filters, TypeScript
sidecars, GitHub content checks) need a language-agnostic contract,
and a generated JSON Schema would invert the dependency.

### 2. Argparse-only CLI, no click

The verb surface is small (`new`, `status`, `render`,
`ledger {list,record}`). An argparse implementation is ~200 lines and
adds no install footprint. Click would mean one more dependency for
zero gained ergonomics at this scale.

**Revisit when:** verbs exceed ~8, or when sub-sub-command nesting
becomes common (`ledger score by-position`-style trees).

### 3. Append-only JSONL ledger at `data/ledger/runs.jsonl`

One JSON object per line. `pre_mortem_ledger.ledger.append_ledger`
refuses to write a row whose `run_id` already exists, and refuses a
second row with the same `(month, run_type)` pair. Corrections land as
a `calibration` row, not as an in-place edit.

**Rejected:** SQLite or DuckDB. The row count is on the order of
twelve per year. JSONL diffs cleanly in git, and the meta-record is
deliberately readable without tooling.

### 4. Brier scoring helpers ship in v0.1; back-scoring wires up in spec 0003

`pre_mortem_ledger.score` exposes pure functions (`brier_score`,
`rank_to_prior`, `pairs_from_premortem`, `brier_for_month`). The
`prior_run_brier` field on the ledger row is `null` in v0.1 because
the first month has no prior data to compare against. The CLI verb
that fills the field (`premortem ledger score`) lands in spec 0003
once the 2026-M07 → 2026-M08 transition produces the first real
status_log entries.

**Rationale:** shipping the scoring math without the wiring lets
spec 0003 land as a thin CLI change instead of a behavioral change to
the ledger contract.

### 5. uv + `[dependency-groups]` + `[tool.uv] package = true`

Dev deps under `[dependency-groups]`, not
`[project.optional-dependencies]`. `[tool.uv]` sets `package = true`
so the project itself is editable-installed into the venv on
`python -m uv sync`. Without these two blocks, `python -m uv run
pytest` falls back to system Python and `from pre_mortem_ledger
import schema` raises `ModuleNotFoundError`.

### 6. STATUS.md sections are a contract

The three H2 sections — `## Current state`, `## Known limits`,
`## Next feature queue` — are matched literally by the factory's
contract gate. Renaming them silently breaks the next factory run for
this repo. R-PML-109 records this constraint.

## Consequences

- The v0.1 row in `data/ledger/runs.jsonl` is the only ledger row.
  Every future run appends.
- The next factory run reads `STATUS.md`'s "Next feature queue" as the
  inbox; the top item is the CI workflow that wires the four gate
  scripts.
- Spec 0003 has a narrow surface: wire `pre_mortem_ledger.score` into
  a new `premortem ledger score` verb, fill `prior_run_brier`, and
  add the structural anti-pattern checks to `voice_lint`.
